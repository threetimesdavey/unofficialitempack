## Purpose of this notice

I, DaveTheBrave, created this repository because I believe in
**BG2:EE Unofficial Item Pack**, value what it adds to BG2:EE, and do not want
the project to disappear or remain inaccessible while players still use it and
wish to contribute.

The word **inherit** is used here only in a community-maintenance sense.

It means accepting practical responsibility for preservation, organization,
bug fixing, documentation, translation integration, and community
coordination.

It does not mean that I created the original work, acquired its copyright, or
replaced Edvin as its historical author.

## Original authorship

BG2:EE Unofficial Item Pack was created by **Edvin**.

Edvin remains the original author and must continue to receive clear and
prominent credit in:

- the repository;
- the readme;
- release notes;
- mod metadata;
- installer text where practical;
- community announcements;
- derivative maintenance releases.

Nothing in this repository should erase, obscure, minimize, or rewrite that
authorship.

The Git history of the community repository begins after years of work had
already been completed by Edvin. Git commit history alone therefore cannot
represent the full authorship history of the project.

## Why community maintenance was started

The original project attracted continued interest because it gives overlooked,
underused, and quest-related BG2:EE items new gameplay value.

The original Beamdog discussion remained active after the latest surviving
release.

Relevant events include:

- the latest surviving original package being identified as version `2.7b`;
- the original author restoring or repairing a download mirror in July 2021;
- an Italian translation being published by DaveTheBrave in September 2024;
- community requests for GitHub hosting and pull-request-based collaboration;
- later reports of missing resources, incorrect restrictions, and mod
  compatibility problems;
- continued player interest in preserving and improving the project.

The purpose of this repository is to give that activity a stable and
transparent home.

## Goals of the maintenance project

The community-maintained continuation aims to:

- preserve the latest surviving original release;
- document the project’s history accurately;
- retain Edvin’s authorship and original design intent;
- integrate authorized translations;
- correct reproducible bugs;
- restore missing required resources where redistribution is permitted;
- replace compatibility-hostile overwrites with targeted WeiDU patches;
- improve compatibility with BG2:EE, EET, and other actively maintained mods;
- preserve stable component numbers where possible;
- provide a public issue tracker;
- review changes through pull requests;
- maintain a human-readable changelog;
- publish reproducible and documented releases;
- welcome translators, testers, modders, and documentation contributors;
- prevent the project from becoming dependent on one unavailable download
  location.

## What community maintenance does not mean

This project does not claim that:

- inactivity transfers copyright;
- absence constitutes permission to relicense a work;
- the original author has surrendered ownership;
- the community maintainer created the original mod;
- every proposed change is automatically part of Edvin’s design;
- public GitHub hosting automatically makes every file open source;
- community demand overrides the rights of an original author;
- historical files may be redistributed without regard to their applicable
  permissions.

The community continuation must distinguish clearly between:

1. Edvin’s original work;
2. later community changes;
3. third-party contributions;
4. game assets owned by their respective rights holders.

## Current stewardship

Until another arrangement is agreed, DaveTheBrave acts as the repository’s
community steward.

The steward’s responsibilities include:

- protecting the project’s history;
- keeping authorship visible;
- reviewing issues and pull requests;
- preventing undocumented or destructive changes;
- coordinating releases;
- recording contributor credit;
- documenting known compatibility risks;
- distinguishing confirmed fixes from planned work;
- responding to the original author in good faith;
- avoiding claims of ownership that have not been granted.

Stewardship is a responsibility, not a claim of creative priority over the
original author.

## Decision-making while the original author is absent

Routine maintenance may proceed when it preserves the original mod’s purpose
and does not depend on a disputed ownership claim.

Routine maintenance includes:

- documentation corrections;
- translation integration;
- reproducible bug fixes;
- restoration of demonstrably missing dependencies;
- safer WeiDU implementation;
- compatibility patches;
- packaging corrections;
- test infrastructure;
- accessibility and readability improvements.

Substantial changes should begin with a public issue.

Substantial changes include:

- redesigning an item;
- significantly changing balance;
- replacing original writing;
- adding major new components;
- changing the project name;
- changing component numbers;
- changing licensing;
- removing original content;
- declaring a new canonical direction for the mod.

When consensus is unclear, compatibility and preservation should take priority
over expansion.

## If Edvin returns

If Edvin returns and wishes to discuss the project, that contact will be treated
as a priority and handled respectfully.

Reasonable verification may be requested through an established public account,
such as the Beamdog account historically associated with the mod, to protect
the project from impersonation.

The first objective will be to understand what Edvin wants.

Possible arrangements include:

### 1. Edvin resumes sole project leadership

If Edvin wishes to take the project back and resume active maintenance, I will
offer to transfer day-to-day leadership and, where technically possible and
appropriate, repository administration or ownership.

The transition should preserve:

- Git history;
- issue history;
- contributor credits;
- release history;
- community fixes;
- translations;
- documentary evidence of the original and community eras.

### 2. Co-maintenance

If Edvin prefers shared maintenance, repository permissions and review
responsibilities can be reorganized so that Edvin and community maintainers
work together.

Edvin’s preferred role should be documented publicly.

### 3. Original-author oversight with community implementation

Edvin may choose to retain creative direction or release approval while
community contributors implement and test changes.

### 4. Repository transfer

If both sides agree, the repository may be transferred to:

- Edvin’s GitHub account;
- an organization selected by Edvin;
- a neutral community organization;
- another agreed maintainer.

A transfer should include a public notice so users understand that the official
maintenance location has changed.

### 5. Separate community fork

If Edvin wishes to maintain a separate original-author version, this repository
may become a clearly labeled community fork.

In that case:

- the distinction between projects will be made explicit;
- neither repository will misrepresent the other;
- compatible fixes may be shared where permissions allow;
- users will be directed to the appropriate project for each release line.

### 6. Archival of the community continuation

If Edvin makes a legitimate request that this continuation stop presenting
itself as the primary maintenance project, the repository may be archived,
renamed, or redirected according to an agreed resolution.

Archival should preserve lawful historical information, contributor credit,
and a pointer to the active project.

## Conduct during a return or ownership discussion

During any discussion with the original author:

- disputed claims should not be presented as settled facts;
- inflammatory community pressure should be discouraged;
- private correspondence should remain private unless all parties agree to
  publication;
- releases directly affected by a rights dispute may be paused;
- existing users should receive a clear public status notice;
- no contributor should be erased from history;
- technical and legal questions should be separated from personal conflict.

The goal is continuity, not possession.

## Rights and licensing status

At the time this notice was prepared, no explicit public license for the
original package had been identified in the original forum post.

Accordingly:

- this notice does not apply a new license to Edvin’s original files;
- a GitHub upload must not be interpreted as a copyright transfer;
- no maintainer may unilaterally place Edvin’s work under MIT, GPL, or another
  license without the required authority;
- contributors must have the right to submit their own work;
- third-party assets must retain their applicable notices;
- game assets must not be redistributed merely because they are technically
  accessible;
- permission received from the original author should be preserved in a stable
  written record.

If Edvin grants explicit maintenance, redistribution, or relicensing
permission, the repository should document:

- who granted permission;
- the date;
- the scope of permission;
- whether modification is permitted;
- whether redistribution is permitted;
- whether commercial use is prohibited or allowed;
- required attribution;
- the chosen license, if any;
- any conditions relating to project leadership or repository transfer.

A future `LICENSE` or `PERMISSION.md` file should not contradict this record.

## Community-created material

New community-created code, writing, translations, tests, and documentation
remain attributable to their respective contributors unless another written
agreement applies.

Accepted contributions may only be distributed according to the repository’s
documented rights and permission framework.

Contributors should not assume that contributing a fix grants permission to
relicense unrelated original material.

## Preservation policy

The repository should preserve, where legally permitted:

- an identifiable historical `2.7b` baseline;
- checksums for original archives;
- original readmes and release notes;
- source files;
- community patches;
- translations;
- changelog history;
- contributor records;
- release artifacts;
- migration notices.

Historical files should be marked clearly so that users can distinguish them
from current community releases.

Original archives should not be silently modified and presented as untouched
historical packages.

## Release naming

Until the project’s permission and versioning policy is settled:

- historical version names remain unchanged;
- version `2.7b` refers to the surviving original baseline;
- community test builds should be labeled clearly as test or pre-release
  builds;
- no release should be called an official Edvin release unless Edvin approves
  that description;
- terms such as **Community Edition** should be used only when their meaning is
  explained;
- release notes should identify which changes are original and which are
  community-created.

## Transparency policy

Important maintenance decisions should be recorded in public issues, pull
requests, release notes, or repository documents whenever privacy and security
permit.

This includes:

- major design changes;
- licensing decisions;
- repository transfers;
- changes to lead maintenance;
- removal of original content;
- incompatibility declarations;
- decisions to archive the project.

Private discussion may be necessary, but the resulting public project status
should be documented clearly.

## Contacting the project

Community questions, bug reports, compatibility reports, and contribution
proposals should be opened through the GitHub issue tracker.

Edvin may contact the maintainer through:

- the GitHub account associated with this repository;
- the original Beamdog forum discussion;
- another verifiable channel chosen by Edvin.
